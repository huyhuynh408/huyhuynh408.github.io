Basic Regular Expression

March 25, 2015

I was interested in learning regular expression(regexp) because I had seen it mentioned before in our coursework that we shouldn't be using regexp yet and saw some examples of it on Stack Overflow. It looked very foreign and in my research, it is still quite a mystery but at least I have a good idea what it is now. To start, my understanding of regexp is a way to match patterns in strings. In layman's terms it means that if I had a sentence like "This is an example sentence I though up", I try to find various patterns like say I wanted to find how many times the letter "A" appears or if there are any numbers in that string.

To understand regexp, we have to learn the syntax. The syntax is quite easy, you will see that regexp begins with a forward slash and ends with a forward slash. A common method used with regexp is match. Match can be written as the operator #match or =~. The match method lets you see if the regexp matches a string. An example would be if I was trying to see if a string contained the letter "Y", the regexp would be /y/. So that means that I could write /y/ =~ "Huy" and the result would return => 1. I could also write /huy/.match("huy") and the result would be #<MatchData "Huy">. If it did not find a match, it would return nil.

Regexp can be a very powerful tool if used correctly. It really cuts down on work and makes it easy to be a good lazy programmer. One example I learned is that I can search a string to see if it contains a number. This would be useful in situations where you asked for someone's name, you don't want them entering numbers. My name would not be "Huy99". Maybe it would, but it is not likely. If I wanted to see if "Huy99" contained numbers, I can write "Huy99".match(/[0-9]/). The result would be #<MatchData 1>. I didn't mention it before, but the [0-9] inside the regexp means that it is looking for any digits from 0 to 9. It could also be written [123456789]] but that is a more tedious way of doing it. Can you guess what [a-z] is looking for? If you said any lowercase letter from a to z you would be correct.

Another thing you can do with regexp is finding and replacing values in your string. Let's say I had a string "Hi Mom!" and I wanted to replace the first two characters to be "bye", I can use regexp to find the first two characters at the beginning of the line. It would look like this: /^../. We are seeing two new things here: 1. the first is the ^. All this means is only look for a match at the beginning of the line and 2. the two dots. The two dots represents 2 characters. Those characters could be a letters, numbers, special characters or even whitespace. Now we can use regexp to find the first two characters at the beginning of our line, we can use the method #sub to make the replacement. Our code would be "Hi Mom!".sub(/^../, "Bye"). That would return "Bye Mom!"

I only showed you two basic examples, but with the right use, they can be extremely powerful tools in your arsenal. Think about what you want and use regexp to get it done more efficiently!

Class Methods

March 13, 2015

What exactly is a class method? In the most basic terms, a class method is a method that is a method that is called on a class. In contrast, an instance method, is a method that is called on an instance of that class. Let's look at a quick example.

Class method:

class Example def self.test puts "I am a class method" end end Example.test

Instance method:

class Example def test puts "I an instance method" end end example = Example.new example.test

Hopefully by looking at this example, it will clear up some confusion. You might have noticed the differnce between the two if you had a keen eye. There are two takeaways I want to point out. The first is that the class method is written differently; the other is that the method test is called directly on the class.

So how do you write a class method? By looking at our example above, we see that one way to do this is to use the syntax self.method_name. Another way is to use the syntax class << self before you write your method. It would look like this:

class Example class << self def test puts "I am a class method" end end end

The other difference we learned about class method is that they are called directly on our class. In my example above, you can see that the class method test was called on the class Example. In the instance method, you had to instantiate the class first and then call the method on your instantiated class. This basically means that class methods are called on classes, instance methods are called on instances.

Ruby Classes

March 7, 2015

Classes in Ruby are like the blueprints. They tell your object what to do when you call it. I will use an example of a die to show you what I mean. Let's think about what a die needs to do. To start, a regular die has six sides; numbers 1-6. When you pick it up and roll it, a random side will land upright. So that's it. The basics of a die is that it starts off with one side up and when you roll it, a random side will land faceup.

class Die def initialize @sides = sides end def sides @sides end def roll rand(1..6) end end

die = Die.new die.roll
dice = [Die.new, Die.new] dice.each { |die| puts die.roll }

Now this is an extremely basic example but it does allow us to examine the basics of Classes. To start off, Classes in Ruby begin with a capital letter. You may have noticed that there is a way to create an Array like this: array = Array.new or creating a Hash like this: hash = Hash.new. Notice that the letter "A" and "H" both have to be capitalized. That's because those are Classes. You could do the same for Strings and Integers but it is a lot easier just to type in string = "This is a new string" or num = 12121.

Next we see that there is an initialize method. The method is not necessary but will be a method that runs automatically when you call on a Class. A good way to think of it in this situation is that you cannot have a die that doesn't have any sides. A die resting will be on one side or the other, that's why it makes sense that sides would be initialized. Below that, you can see that there is a method for sides that just returns the instance variable sides. This could be done with the attr_reader method but I won't go into that here.

Here we finally come to some good stuff in our class. This roll method is what we have been waiting for. It allows us to roll the die and return a random number between 1 and 6. You may not be familiar with rand() but it is quite simple. When you put an argument into rand, it will return a random number between 0 and the number before that argument. I used the range 1..6 instead to make it cleaner to see. This acts just like how we expect a normal die would.

Okay, now that we have our Class, what can we do with it? Well looking on the next line, you can see where I created a new object called die and set it equal to a new Die class. This is just like creating those Arrays and Hashes above. By setting the die to Die.new, I am creating a new instance of that object. This is what is called instantiation. Now with this new object, I can finally do what this Class was made to do. By now using die.roll I am now using the method I just created. This roll will now generate a random number as explained earlier.

What if we wanted dice instead of a die? Well you can see in the last part of the code that I have done just that. I made this new object called dice out of an Array of two Die classes. When dice is called, it will return two dice with two random numbers just like we expect. To do this, we need to iterate through the array. This may seem more complex, but it really just means that the code is run through each part of the Array and does what we tell it, in this case we tell it to use the roll method on each part of the Array.

Enumerable#cycle

March 1, 2015

I chose the Enumerable#cycle because I was not familiar with this method and it would be nice to research what it. Here is what Ruby Docs has to say about this method.

cycle(n=nil) { |obj| block } → nil
cycle(n=nil) → an_enumerator

Calls block for each element of enum repeatedly n times or forever if none or nil is given. If a non-positive number is given or the collection is empty, does nothing. Returns nil if the loop has finished without getting interrupted.

#cycle saves elements in an internal array so changes to enum after the first pass have no effect. If no block is given, an enumerator is returned instead.

So what does this even mean in regular English? Simply put, it allows you to call a block on your object a specific number of times, or if you don't give it a number, it will continue looping the block forever. Let's look at some examples:

a = ["a", "b", "c"]
a.cycle { |x| puts x } # print, a, b, c, a, b, c,.. forever.
a.cycle(2) { |x| puts x } # print, a, b, c, a, b, c.

Looking at the second line, we can see that if we use the cycle method with the block of { |x| puts x }, the array will keep printing on forever. On the third line, we see that by using the argument of 2, the array only prints twice and stops.

Hashes v. Arrays

February 22, 2015

Today I learned more about hashes and arrays. The two are very similar but also quite different. Each has its use and different syntaxes. The easiest way to tell the difference between an array and a hash is the key/value pairs. In arrays, the key is an integer. In hashes the key is any object like a string or integer. The value in arrays and hashes could be almost anything you want, they could easily be strings, integers, floats, other arrays, etc. Let's take a quick look at the syntax for both.

Arrays are written with brackets like this: []. Hashes are written with braces like this: {}.

An example of an array would be [1, 'Bob', 'Chicago'].

The same hash would be {:one => 1, :name => 'Bob', :city => 'Chicago'}.

Now I didn't mention it earlier but you can see two big differences from the hash example. The first is the :. The colon denotes that whatever comes after it is the key to whatever comes after the =>. What does that even mean? It is pretty simple, the first part is the key and the second is the value. You can think of it sort of like a dictionary. If you look up the word apple in the dictionary, the word apple would be the key and the definition of apple would be the value. The same applies to arrays but their values are already predefined as integers. The key to the number 1 in the first array example is 0, the key to number 2 is 1, etc. The keys/indexes for arrays begin at 0 and then 1 for the next value, then 2, etc. I have no idea why indexes start at 0 but that's the way most computer languages have it written.

To make it more clear, let's see how we can call back the values from arrays and hashes. To call the name Bob from an array, we would need to find it's place in the index. Since it is in the second place it has an index key of 1. Then to call Bob in an array we would type [1]. To do the same in a hash, we would need to find the key. I have conveniently named the key for Bob as name. To call Bob from a hash we would type ['name'].

As you can see from the example, there are some benefits and drawbacks of each method. Arrays would be better for an ordered lists that you don't need to remember specific keys to. Hashes on the other hand would be better for lists that you would want to know what keys are needed to access them. The difference would be like if you wanted a shopping list and wanted to know which was the first item on the list and a dicitionary where you want to know what word to look up to find its definition.

Cascading style sheet has a a lot of ways to help you display elements properly on your website. One of the most useful feature is inline and inline-block. This allows you to have elements on the same line instead of on different rows. Let's see how it looks and then we will go into the difference between inline v. inline-block.

The biggest difference between inline-block and inline is the block attribute in inline-block. Inline-block still keeps the elements on the same line as inline but it also acts just like the block display. It allows the elements to retain their height and width attributes. In the inline picture, all of the divs have the same shape because they are limited by the text they contain. They are only as big as whatever is in them. In inline-block, you can specify those attributes even if most of it is empty.

Version Control

February 04, 2015

There are two great benefits to version control: collaboration with others and multiple saved files. Version control (we are learning Git but it applies to all version control software) allows a very large number of individuals to work on the same file at the same time. It does this by having a master file from which you can create branches that you can play around with it. When you are satisfied with the results, the changes are merged with the master. It will find the parts that were changed and only modify those. That means that you can be working on one part of the file while your partner works on another. Then you can come back to combine the work you have done.

In case there were any issues, you can revert back to an older master file. That is the other benefit of version control. It allows you to see all of the files that you have commited (saved). It works very very similarly to a save point in a game. You can proceed forward but if you ever wanted to return to a checkpoint you always can. If for example, you were working on a branch (a copy of the master file) and were having bugs that you couldn't figure out, it may be beneficial to start from scratch. In that instance you still have the master file without having to manually unravel your mess. It also works great in situations where you want to see why certain code was typed the way they were or if you wanted to see who made those changes. These records are always available and makes your life easier in case you had questions.

In DBC we are using GitHub to store our code. GitHub is similar to Git in that it shares the same great benefits but it is online. Being online allows anyone access from any computer. I can download the files I need and when I am done, I simply upload them to GitHub for storage. Then any of your teamates can do the same. It allows you to easily share the work that you do with the rest of the world.
